# 技术文档写作模式与模板

## 概述

本文档总结了技术文档的标准结构、常用表达模式和写作规范，为技术文档的学习和创作提供参考。

## 技术文档标准结构

### 通用文档结构

```markdown
# 文档标题

## 概述
- 简要说明文档内容
- 适用对象
- 前置条件

## 背景/简介
- 相关背景信息
- 问题域说明
- 基本概念

## 核心内容
### 主题 1
### 主题 2
### 主题 3

## 实践指南
- 使用方法
- 示例代码
- 最佳实践

## 常见问题
- FAQ
- 故障排除
- 注意事项

## 参考资料
- 相关链接
- 延伸阅读
- 参考文档
```

### API 文档结构

```markdown
# API 名称

## 概述
- 功能说明
- 版本信息
- 基础 URL

## 认证
- 认证方式
- 获取凭证
- 使用说明

## 端点列表

### [方法] /path/to/endpoint

**描述：** [简要描述]

**请求参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| param1 | string | 是 | 参数说明 |
| param2 | number | 否 | 参数说明 |

**请求示例：**

```http
GET /api/endpoint?param1=value
```

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

**响应字段：**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| code | number | 状态码 |
| message | string | 响应消息 |
| data | object | 响应数据 |

## 错误码
- 错误码列表
- 错误说明
- 处理建议
```

## 常用表达模式

### 指令性表达

**直接命令：**
- "点击..."
- "选择..."
- "输入..."
- "执行..."

**祈使句：**
- "请确保..."
- "务必..."
- "建议..."
- "必须..."

**条件指令：**
- "如果...，则..."
- "当...时..."
- "在...情况下..."

### 说明性表达

**定义概念：**
- "...是指..."
- "...定义为..."
- "...包括..."

**描述功能：**
- "此功能用于..."
- "该选项允许..."
- "本模块提供..."

**解释原因：**
- "这是因为..."
- "目的是为了..."
- "这样做的理由是..."

### 警告性表达

**注意事项：**
- "注意："
- "警告："
- "重要："
- "切记："

**限制说明：**
- "不支持..."
- "仅适用于..."
- "限制为..."

**风险提示：**
- "可能导致..."
- "避免..."
- "请勿..."

## 术语使用规范

### 技术术语

**一致性原则：**
- 同一概念使用统一术语
- 首次出现时给出定义
- 保持术语翻译一致

**示例：**
- API / Application Programming Interface
- SDK / Software Development Kit
- Endpoint / 端点
- Payload / 负载

### 操作术语

**常用动词：**
- 配置（configure）
- 部署（deploy）
- 安装（install）
- 集成（integrate）
- 调用（invoke/call）
- 返回（return）
- 处理（process）
- 验证（validate）

**标准搭配：**
- 执行命令
- 调用接口
- 返回结果
- 处理数据
- 验证身份

### 状态术语

**常见状态：**
- 成功（success）
- 失败（failure）
- 进行中（pending/in progress）
- 已完成（completed）
- 已取消（cancelled）

**状态描述：**
- "操作成功"
- "执行失败"
- "处理中"
- "已完成"

## 格式和样式指南

### 标题层级

```markdown
# 一级标题（文档标题）
## 二级标题（主要章节）
### 三级标题（子章节）
#### 四级标题（细节说明）
```

**使用原则：**
- 层级清晰，逻辑分明
- 不跳级使用
- 同级内容并列

### 代码块

**行内代码：**
使用 `单个反引号` 包裹

**代码块：**
使用三个反引号包裹，指定语言

\```javascript
function example() {
  // code here
}
\```

**命令行：**
\```bash
npm install package-name
\```

### 强调和重点

**加粗强调：**
- **重要概念**
- **关键步骤**

**斜体标注：**
- *术语定义*
- *注意事项*

**代码字体：**
- 使用等宽字体表示变量、参数名

### 列表使用

**无序列表：**
- 要点 1
- 要点 2
  - 子要点
  - 子要点

**有序列表：**
1. 步骤一
2. 步骤二
3. 步骤三

**任务列表：**
- [ ] 待完成任务
- [x] 已完成任务

### 表格格式

```markdown
| 列 1 | 列 2 | 列 3 |
|------|------|------|
| 数据 | 数据 | 数据 |
| 数据 | 数据 | 数据 |
```

**对齐方式：**
| 左对齐 | 居中对齐 | 右对齐 |
|:-------|:--------:|-------:|
| 内容   |   内容   |   内容 |

## 写作风格规范

### 语言风格

**技术文档特点：**
- 客观、准确
- 简洁、明了
- 逻辑、严谨
- 专业、规范

**语言要求：**
- 使用简单句
- 避免复杂从句
- 使用主动语态
- 避免歧义表达

### 时态使用

**一般现在时：**
- 描述功能特性
- 说明操作步骤
- 解释概念原理

**示例：**
- "该功能支持..."
- "点击按钮后..."
- "系统会自动..."

### 人称使用

**第二人称（你）：**
- 操作指南
- 用户手册
- 教程文档

**第三人称（它/系统）：**
- 功能描述
- 系统说明
- 技术规范

### 简洁性原则

**避免冗余：**
- ❌ "用户可以点击按钮来..."
- ✅ "点击按钮..."

**去除废话：**
- ❌ "基本上，这个功能是用来..."
- ✅ "该功能用于..."

**直接表达：**
- ❌ "在这个阶段，我们需要..."
- ✅ "此时..."

## 最佳实践

### 1. 受众导向

**明确读者：**
- 开发者 → 技术细节、代码示例
- 产品经理 → 功能概述、使用场景
- 最终用户 → 操作指南、注意事项

**调整深度：**
- 根据读者背景调整技术深度
- 提供适当的背景说明
- 使用读者熟悉的术语

### 2. 结构化组织

**层次清晰：**
- 使用标题和子标题
- 合理分组相关内容
- 提供目录导航

**逻辑连贯：**
- 内容顺序符合认知逻辑
- 步骤之间有自然过渡
- 前后有呼应

### 3. 示例驱动

**代码示例：**
- 提供完整可运行的示例
- 包含必要的注释
- 展示最佳实践

**场景示例：**
- 真实使用场景
- 常见用例
- 边界情况

### 4. 可维护性

**版本管理：**
- 标注文档版本
- 记录更新历史
- 关联产品版本

**易于更新：**
- 模块化内容
- 避免重复信息
- 统一格式规范

## 常见问题与解决方案

### 问题 1：文档过于冗长

**原因：** 包含过多细节或重复信息

**解决：**
- 精简表达
- 移除冗余内容
- 使用引用代替重复

### 问题 2：结构混乱

**原因：** 缺少整体规划

**解决：**
- 制定大纲
- 明确章节目标
- 调整内容顺序

### 问题 3：表达不清晰

**原因：** 缺少示例或说明

**解决：**
- 添加示例
- 补充说明
- 使用图示

### 问题 4：难以维护

**原因：** 缺少模块化

**解决：**
- 拆分内容
- 建立引用
- 版本控制
