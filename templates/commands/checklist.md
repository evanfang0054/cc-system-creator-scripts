## 检查清单目的："英文编写的单元测试"

**核心概念**：检查清单是**需求编写的单元测试** - 它们验证给定领域中需求的质量、清晰度和完整性。

**不用于验证/测试**：

- ❌ 不用于"验证按钮点击正确"
- ❌ 不用于"测试错误处理有效"
- ❌ 不用于"确认 API 返回 200"
- ❌ 不用于检查代码/实现是否符合规范

**用于需求质量验证**：

- ✅ "是否为所有卡片类型定义了视觉层次需求？"（完整性）
- ✅ "'突出显示'是否通过具体的尺寸/位置进行了量化？"（清晰度）
- ✅ "所有交互元素的悬停状态需求是否一致？"（一致性）
- ✅ "是否为键盘导航定义了无障碍需求？"（覆盖范围）
- ✅ "规范是否定义了 logo 图像加载失败时的处理方式？"（边界情况）

**比喻**：如果你的规范是用英文编写的代码，那么检查清单就是它的单元测试套件。你正在测试需求是否编写良好、完整、无歧义并准备好实施 - 而不是实现是否有效。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 执行步骤

1. **设置**：从仓库根目录运行 `{SCRIPT}` 并解析 JSON 以获取 FEATURE_DIR 和 AVAILABLE_DOCS 列表。
   - 所有文件路径必须是绝对路径。
   - 对于参数中的单引号，如 "I'm Groot"，使用转义语法：e.g 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

2. **澄清意图（动态）**：根据用户表述 + 从规范/计划/任务中提取的信号，推导最多三个初始上下文澄清问题（无预编目录）。它们必须：
   - 从用户的表述中生成 + 从规范/计划/任务中提取的信号
   - 仅询问实质性改变检查清单内容的信息
   - 如果在 `$ARGUMENTS` 中已经明确，则单独跳过
   - 优先考虑精度而非广度

   生成算法：
   1. 提取信号：功能领域关键词（例如 auth, latency, UX, API）、风险指标（"critical", "must", "compliance"）、利益相关者提示（"QA", "review", "security team"）和明确的可交付成果（"a11y", "rollback", "contracts"）。
   2. 将信号聚类为候选关注领域（最多 4 个），按相关性排序。
   3. 识别可能的受众和时机（作者、审阅者、QA、发布）（如果未明确）。
   4. 检测缺失的维度：范围广度、深度/严格度、风险重点、排除边界、可测量的验收标准。
   5. 从这些原型中制定问题：
      - 范围细化（例如，"这应该包括与 X 和 Y 的集成接触点，还是仅限于本地模块正确性？"）
      - 风险优先级（例如，"这些潜在风险领域中有哪些应该接受强制性关卡检查？"）
      - 深度校准（例如，"这是一个轻量级的预提交合理性清单，还是正式的发布关卡？"）
      - 受众框架（例如，"这仅由作者使用，还是在 PR 审查期间由同行使用？"）
      - 边界排除（例如，"我们应该在本轮中明确排除性能调优项目吗？"）
      - 场景类别差距（例如，"未检测到恢复流程——回滚/部分失败路径是否在范围内？"）

   问题格式规则：
   - 如果提供选项，生成一个紧凑的表格，列：选项 | 候选 | 为什么重要
   - 最多限制 A-E 选项；如果自由形式答案更清晰，则省略表格
   - 从不让用户重述他们已经说过的内容
   - 避免推测性类别（无幻觉）。如果不确定，明确询问："确认 X 是否属于范围。"

   无法交互时的默认值：
   - 深度：标准
   - 受众：审阅者（PR）（如果与代码相关）；否则为作者
   - 重点：前 2 个相关性聚类

   输出问题（标记为 Q1/Q2/Q3）。回答后：如果 ≥2 个场景类别（Alternate / Exception / Recovery / Non-Functional 领域）仍不清楚，你可以再问最多两个有针对性的后续问题（Q4/Q5），每个问题都有单行理由（例如，"未解决的恢复路径风险"）。不要超过五个问题。如果用户明确拒绝更多，则跳过升级。

3. **理解用户请求**：结合 `$ARGUMENTS` + 澄清答案：
   - 推导检查清单主题（例如 security, review, deploy, ux）
   - 整合用户提到的明确必需项目
   - 将重点选择映射到类别脚手架
   - 从规范/计划/任务中推断任何缺失的上下文（不要产生幻觉）

4. **加载功能上下文**：从 FEATURE_DIR 读取：
   - spec.md：功能需求和范围
   - plan.md（如果存在）：技术细节、依赖关系
   - tasks.md（如果存在）：实施任务

   **上下文加载策略**：
   - 仅加载与活动关注领域相关的必要部分（避免完整文件转储）
   - 优先将长部分总结为简洁的场景/需求要点
   - 使用渐进式披露：仅在检测到差距时添加后续检索
   - 如果源文档很大，生成临时摘要项，而不是嵌入原始文本

5. **生成检查清单** - 创建"需求的单元测试"：
   - 创建 `FEATURE_DIR/checklists/` 目录（如果不存在）
   - 生成唯一的检查清单文件名：
     - 使用基于领域的简短描述性名称（例如 `ux.md`、`api.md`、`security.md`）
     - 格式：`[domain].md`
     - 如果文件存在，追加到现有文件
   - 从 CHK001 开始按顺序编号项目
   - 每次 `/speckit.checklist` 运行都会创建一个新文件（从不覆盖现有检查清单）

   **核心原则 - 测试需求，而非实现**：
   每个检查清单项目必须评估需求本身是否：
   - **完整性**：所有必要的需求是否存在？
   - **清晰度**：需求是否明确无歧义？
   - **一致性**：需求是否相互一致？
   - **可测量性**：需求能否客观验证？
   - **覆盖范围**：是否解决了所有场景/边界情况？

   **类别结构** - 按需求质量维度分组：
   - **需求完整性**（是否记录了所有必要的需求？）
   - **需求清晰度**（需求是否明确无歧义？）
   - **需求一致性**（需求是否一致且无冲突？）
   - **验收标准质量**（成功标准是否可测量？）
   - **场景覆盖**（是否解决了所有流程/情况？）
   - **边界情况覆盖**（是否定义了边界条件？）
   - **非功能需求**（性能、安全性、无障碍性等 - 是否已指定？）
   - **依赖关系和假设**（是否已记录和验证？）
   - **歧义和冲突**（需要澄清什么？）

   **如何编写检查清单项目 - "英文编写的单元测试"**：

   ❌ **错误**（测试实现）：
   - "验证着陆页显示 3 个剧集卡片"
   - "测试桌面上的悬停状态有效"
   - "确认 logo 点击导航到主页"

   ✅ **正确**（测试需求质量）：
   - "是否明确指定了特色剧集的确切数量和布局？" [完整性]
   - "'突出显示'是否通过具体的尺寸/位置进行了量化？" [清晰度]
   - "所有交互元素的悬停状态需求是否一致？" [一致性]
   - "是否为所有交互式 UI 定义了键盘导航需求？" [覆盖范围]
   - "是否指定了 logo 图像加载失败时的回退行为？" [边界情况]
   - "是否为异步剧集数据定义了加载状态需求？" [完整性]
   - "规范是否为竞争的 UI 元素定义了视觉层次？" [清晰度]

   **项目结构**：
   每个项目应遵循此模式：
   - 询问需求质量的问题格式
   - 专注于规范/计划中编写的内容（或未编写的内容）
   - 在括号中包含质量维度 [Completeness/Clarity/Consistency/etc.]
   - 检查现有需求时引用规范章节 `[Spec §X.Y]`
   - 检查缺失需求时使用 `[Gap]` 标记

   **按质量维度的示例**：

   完整性：
   - "是否为所有 API 失败模式定义了错误处理需求？[Gap]"
   - "是否为所有交互元素指定了无障碍需求？[Completeness]"
   - "是否为响应式布局定义了移动断点需求？[Gap]"

   清晰度：
   - "'快速加载'是否通过具体的时间阈值进行了量化？[Clarity, Spec §NFR-2]"
   - "'相关剧集'的选择标准是否明确定义？[Clarity, Spec §FR-5]"
   - "'突出显示'是否通过可测量的视觉属性定义？[Ambiguity, Spec §FR-4]"

   一致性：
   - "所有页面的导航需求是否一致？[Consistency, Spec §FR-10]"
   - "着陆页和详情页之间的卡片组件需求是否一致？[Consistency]"

   覆盖范围：
   - "是否为零状态场景（无剧集）定义了需求？[Coverage, Edge Case]"
   - "是否解决了并发用户交互场景？[Coverage, Gap]"
   - "是否为部分数据加载失败指定了需求？[Coverage, Exception Flow]"

   可测量性：
   - "视觉层次需求是否可测量/可测试？[Acceptance Criteria, Spec §FR-1]"
   - "'平衡的视觉权重'能否客观验证？[Measurability, Spec §FR-2]"

   **场景分类和覆盖**（需求质量重点）：
   - 检查是否存在以下需求：主要、替代、异常/错误、恢复、非功能场景
   - 对于每个场景类别，询问："[场景类型] 需求是否完整、清晰和一致？"
   - 如果场景类别缺失："[场景类型] 需求是故意排除还是缺失？[Gap]"
   - 发生状态变更时包括恢复/回滚："是否为迁移失败定义了回滚需求？[Gap]"

   **可追溯性需求**：
   - 最低要求：≥80% 的项目必须至少包含一个可追溯性引用
   - 每个项目应引用：规范章节 `[Spec §X.Y]`，或使用标记：`[Gap]`、`[Ambiguity]`、`[Conflict]`、`[Assumption]`
   - 如果不存在 ID 系统："是否建立了需求和验收标准 ID 方案？[Traceability]"

   **发现并解决问题**（需求质量问题）：
   询问关于需求本身的问题：
   - 歧义："术语 'fast' 是否通过具体指标量化？[Ambiguity, Spec §NFR-1]"
   - 冲突："§FR-10 和 §FR-10a 之间的导航需求是否冲突？[Conflict]"
   - 假设："'始终可用的 podcast API' 的假设是否已验证？[Assumption]"
   - 依赖关系：是否记录了外部 podcast API 需求？[Dependency, Gap]
   - 缺失定义："'visual hierarchy' 是否通过可测量的标准定义？[Gap]"

   **内容整合**：
   - 软限制：如果原始候选项目 > 40，按风险/影响优先考虑
   - 合并检查相同需求方面的近似重复项
   - 如果 >5 个低影响的边界情况，创建一个项目："需求中是否解决了边界情况 X、Y、Z？[Coverage]"

   **🚫 绝对禁止** - 这些会使其成为实现测试，而非需求测试：
   - ❌ 任何以 "Verify"、"Test"、"Confirm"、"Check" 开头的项目 + 实现行为
   - ❌ 引用代码执行、用户操作、系统行为
   - ❌ "正确显示"、"正常工作"、"按预期运行"
   - ❌ "点击"、"导航"、"渲染"、"加载"、"执行"
   - ❌ 测试用例、测试计划、QA 程序
   - ❌ 实现细节（框架、API、算法）

   **✅ 必需模式** - 这些测试需求质量：
   - ✅ "是否为 [场景] 定义/指定/记录了 [需求类型]？"
   - ✅ "[模糊术语] 是否通过具体标准量化/澄清？"
   - ✅ "[章节 A] 和 [章节 B] 之间的需求是否一致？"
   - ✅ "[需求] 能否客观测量/验证？"
   - ✅ "需求中是否解决了 [边界情况/场景]？"
   - ✅ "规范是否定义了 [缺失方面]？"

6. **结构参考**：按照 `templates/checklist-template.md` 中的规范模板生成检查清单，包括标题、元章节、类别标题和 ID 格式。如果模板不可用，使用：H1 标题、purpose/created 元行、`##` 类别章节，包含 `- [ ] CHK### <requirement item>` 行，ID 从 CHK001 开始全局递增。

7. **报告**：输出创建的检查清单的完整路径、项目计数，并提醒用户每次运行都会创建一个新文件。总结：
   - 选择的关注领域
   - 深度级别
   - 执行者/时机
   - 任何合并的明确用户指定的必需项目

**重要提示**：每次 `/speckit.checklist` 命令调用都会使用简短、描述性的名称创建检查清单文件，除非文件已存在。这允许：

- 多种不同类型的检查清单（例如 `ux.md`、`test.md`、`security.md`）
- 简单、易记的文件名，指示检查清单目的
- 在 `checklists/` 文件夹中轻松识别和导航

为了避免混乱，请使用描述性类型并在完成后清理过时的检查清单。

## 示例检查清单类型和示例项目

**UX 需求质量**：`ux.md`

示例项目（测试需求，而非实现）：

- "是否通过可测量的标准定义了视觉层次需求？[Clarity, Spec §FR-1]"
- "是否明确指定了 UI 元素的数量和位置？[Completeness, Spec §FR-1]"
- "交互状态需求（hover、focus、active）是否一致定义？[Consistency]"
- "是否为所有交互元素指定了无障碍需求？[Coverage, Gap]"
- "是否定义了图像加载失败时的回退行为？[Edge Case, Gap]"
- "'突出显示'能否客观测量？[Measurability, Spec §FR-4]"

**API 需求质量**：`api.md`

示例项目：

- "是否为所有失败场景指定了错误响应格式？[Completeness]"
- "是否通过具体阈值量化了速率限制需求？[Clarity]"
- "所有端点的身份验证需求是否一致？[Consistency]"
- "是否为外部依赖关系定义了重试/超时需求？[Coverage, Gap]"
- "需求中是否记录了版本控制策略？[Gap]"

**性能需求质量**：`performance.md`

示例项目：

- "是否通过具体指标量化了性能需求？[Clarity]"
- "是否为所有关键用户旅程定义了性能目标？[Coverage]"
- "是否指定了不同负载条件下的性能需求？[Completeness]"
- "性能需求能否客观测量？[Measurability]"
- "是否为高负载场景定义了降级需求？[Edge Case, Gap]"

**安全需求质量**：`security.md`

示例项目：

- "是否为所有受保护的资源指定了身份验证需求？[Coverage]"
- "是否为敏感信息定义了数据保护需求？[Completeness]"
- "威胁模型是否已记录且需求与其一致？[Traceability]"
- "安全需求是否与合规义务一致？[Consistency]"
- "是否定义了安全失败/违规响应需求？[Gap, Exception Flow]"

## 反例：不要做什么

**❌ 错误 - 这些测试实现，而非需求：**

```markdown
- [ ] CHK001 - 验证着陆页显示 3 个剧集卡片 [Spec §FR-001]
- [ ] CHK002 - 测试桌面上的悬停状态正常工作 [Spec §FR-003]
- [ ] CHK003 - 确认 logo 点击导航到主页 [Spec §FR-010]
- [ ] CHK004 - 检查相关剧集部分显示 3-5 个项目 [Spec §FR-005]
```

**✅ 正确 - 这些测试需求质量：**

```markdown
- [ ] CHK001 - 是否明确指定了特色剧集的数量和布局？[Completeness, Spec §FR-001]
- [ ] CHK002 - 是否为所有交互元素一致定义了悬停状态需求？[Consistency, Spec §FR-003]
- [ ] CHK003 - 所有可点击品牌元素的导航需求是否清晰？[Clarity, Spec §FR-010]
- [ ] CHK004 - 是否记录了相关剧集的选择标准？[Gap, Spec §FR-005]
- [ ] CHK005 - 是否为异步剧集数据定义了加载状态需求？[Gap]
- [ ] CHK006 - "visual hierarchy" 需求能否客观测量？[Measurability, Spec §FR-001]
```

**关键区别：**

- 错误：测试系统是否正常工作
- 正确：测试需求是否正确编写
- 错误：验证行为
- 正确：验证需求质量
- 错误："它是否做 X？"
- 正确："X 是否明确指定？"
